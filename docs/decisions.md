# 需求取捨決策

> 這份文件是 Release 01 使用者訪談後的決策草稿。最後仍需要小組刪改與確認，不能把 AI 草稿直接當成最終產品決策。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

在資訊混亂的情況下，v1 優先服務資訊整理者。資訊整理者介於回報者與行動者之間，可以先把不完整、未確認或互相衝突的原始資訊整理清楚，協助判斷哪些資訊需要人工確認、哪些不能直接變成任務。

這樣做可以提升後續分配資源與任務的效率，也能降低行動者根據錯誤資訊採取行動的風險。資訊整理者也能作為溝通橋樑：一方面保留回報者提供的原始脈絡，不把不確定內容改寫成事實；另一方面把整理後的風險、限制與待確認問題交給後續協作者理解。

## 三方需求取捨後的觀點

- 回報者：希望能快速留下不完整資訊，但不能讓轉述或不確定內容被誤認為已確認。
- 資訊整理者：需要把原始資訊、AI 推論與人類判斷分清楚，讓後續資源與任務分配有比較可靠的依據。
- 行動者：最在意哪些資訊可以行動、哪些仍不能用；在查核不足前，不應把行動者作為 v1 的直接使用者。

## 這個使用者最想完成什麼

- 在大量原始資訊中，看出哪些資訊可以整理成候選資料，哪些只能保留為待確認線索。
- 清楚留下原文依據、AI 推論、人類修正與尚未確認的地方。
- 把回報者提供的原始脈絡整理成後續協作者能理解的資訊，但不把未確認內容包裝成事實。
- 協助後續資源與任務分配前，先標出風險、限制與需要人工確認的問題。

## 這個使用者最怕什麼錯誤

- AI 或 UI 把不完整資訊變成看似正式的任務。
- 來源類型被誤解成可信度，例如把截圖或轉述誤讀成官方已確認。
- 後續協作者不知道哪些內容是原文，哪些是推論或整理草稿。

## 我今天暫時不處理的需求

- 暫時不做回報者輸入表單。
- 暫時不做行動者派遣畫面；目前資料仍需要整理與查核，不適合直接支援行動決策。
- 暫時不做自動合併重複資訊。
- 暫時不做地圖、路線、通知、後端或外部查證。

## 我不採用的 AI 建議

- AI 建議：自動把同地點、同物資的回報合併成目前狀態。
  - 不採用原因：時間、來源、查核狀態可能不同，自動合併會把不確定資訊包裝成確定結果。

- AI 建議：把 persona 回饋直接改成 v1 功能清單。
  - 不採用原因：Release 01 目標是需求分析與取捨，不是立即實作；功能清單需要等人類確認後再進入流程設計。

## 必須保留人工確認的判斷

- 是否能把某筆原始資訊視為可行動資訊。
- 是否存在真實地點、當事人同意、隱私或安全風險。
- 不同時間、不同來源的資訊是否能互相覆蓋或合併。
- AI 產生的候選類型、信心程度與下一步建議是否合理。

## 今天明確不做什麼

- 不修改 `src/` 或新增 v1 UI。
- 不新增真實資料、真實地址、電話、人物或地圖資訊。
- 不呼叫外部 API、資料庫、後端或 runtime LLM。
- 不把 release 材料改寫成自己的答案，也不修改 `release-packs/`。

## 我仍不確定的地方

- v1 要保留多少 Phase 0 的草稿編輯能力，才不會讓整理者負擔太重。
- 資訊整理者整理出的結果，要到什麼程度才足以交給行動者參考。
- 小隊討論後是否仍同意優先服務資訊整理者，或需要調整成更重視回報者輸入流程。
