# Phase 0 Observations

## 我們做出了什麼？

- 建立了整理工作台的編輯介面，允許為原始資訊建立、編輯、刪除、重設整理草稿
- 初始化了 6 筆原始資訊（M-001、M-002、M-003、M-006、M-009、M-010）的候選判斷
- 在每筆草稿中記錄候選類型、信心程度、判斷依據、卡住的地方、下一步行動、人工審查備註
- 顯示已驗證 ✓ 的完成項目，以及尚未完成的項目

## 哪些資料讓 agent 很容易整理？

- **M-010 溪畔活動中心 14:35 盤點** - 非常詳細的現場資訊，包括具體數字（雨鞋 12 雙、尺寸 26-28）、收集狀態、下次盤點時間。信心高，適合直接成為地點狀態更新。
- **M-009 光復車站 14:20 集合點回報** - 清楚的現場回報，明確說明限制（只接受已完成報到的志工）、公告位置。信心中等，適合地點狀態更新。
- **M-003 老街口水電需求更新** - 明確表達需求優先順序變化（從鏟子不缺→水電比較需要），雖然有「原本單可能沒更新」的風險，但整體判斷相對直接。

## 哪些資料讓 agent 亂補或誤判？

- **M-005 道路封閉公告** - sourceType 標示為「官方公告」，但原文本身說「截圖寫…」「不知道是哪一天」「也不知道是不是官方」。資訊標籤與內容嚴重不符，agent 可能把來源標籤當成已驗證事實，實際上內文完全是待確認。
- **M-002 vs M-004 雨鞋庫存衝突** - M-002 說「早上有、下午不知道」，M-004 說「有很多」。Agent 可能直接合併成「有雨鞋」，忽略時間敏感性和資訊重複度的問題。應該對應到同一個「溪畔活動中心」地點的不同時間回報，但原資料沒有明確連結。
- **M-006 集合點矛盾** - 同一個地點（學校側門）的兩筆回報直接衝突：「可以當集合點」vs「淹水不適合停留」。Agent 若沒有時間戳判斷順序，容易誤判或隨意選擇一個版本。

## 哪些資訊不能直接變成任務？

- **M-001 清泥需求** - 「老雜貨店後面」不是明確地址，無法定位；「需要十幾個人」是估計還是確認需求未知。需要詢問更精確的地點與人力需求確認。
- **M-002 雨鞋庫存（早上）** - 時間已過期（09:25 回報，距現在已 5+ 小時），應改用 M-010 的 14:35 最新盤點。
- **M-006 學校側門** - 資訊衝突無法判斷當前狀態，不知道「淹水不適合」是完全封閉還是暫時積水。
- **M-008 A 區派人限制** - 「先不要再派人」沒有說是人太多、道路危險，還是任務已完成。無法根據這個資訊做派遣決定。
- **M-011 長者家具搬動需求** - 「尚未確認長者是否同意公開完整地址」，隱私/同意風險。位置描述「大進路口往溪邊方向第二排住家」太模糊。
- **M-012 外地家屬來電藥品需求** - 三級訊息鏈（外地家屬→親友→親友需求），家屬不在現場、無法確認位置、無法確認親友同意。完全無法驗證或行動。

## 哪些資訊的操作者不是當事人？

- **M-011 長者搬家需求** - 「現場志工代一位不方便使用手機的長者轉述」，志工是轉述者而非當事人；長者的實際需求和同意狀態都經過志工過濾。
- **M-012 家屬來電親友需求** - 「外地家屬來電表示親友…」三級轉述，來電者既不是需求者本人也不是現場人員。
- **M-009 志工代回報集合點限制** - 「回報者說入口公告貼在站前遮雨棚」是志工的觀察而非官方同步的公告，存在資訊延遲風險。

## Agent 判斷的侷限

1. **來源標籤與內容不符** - sourceType 有時與原文描述不一致（如 M-005），agent 容易過度信任標籤。
2. **時間敏感性** - M-002、M-004、M-009、M-010 都涉及現場庫存或狀態，且距離回報時間遠近不同。不標注時間距離容易導致過期資訊被誤用。
3. **資訊重複檢測** - M-002 和 M-004 看似在講同一個地點的同一件物資，但 agent 沒有明確機制判別它們的時間順序或優先度。
4. **二手資訊的風險** - M-011、M-012 都涉及轉述或三級鏈條，agent 需要明確標示「需要原當事人確認」。
5. **缺失的上下文** - M-008「先不要再派人」沒有原因說明，agent 無法判斷是成功完成、危險、還是人力過剩。
